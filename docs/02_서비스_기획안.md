# TICKETO — 소규모 공연·클래스 선착순 예매 서비스 · 서비스 기획안 (PRD)

> 풀스택 학습 + 포트폴리오용 사이드 프로젝트 · Next.js + NestJS · 1개월

---

## 1. 배경 & 문제 정의

인기 있는 소규모 공연이나 원데이 클래스는 좌석이 적어서 **오픈과 동시에 몇 초 만에 매진**된다. 이때 서버가 흔히 겪는 문제:

- 동시에 몰린 요청이 같은 좌석을 중복 판매(**초과 판매**)한다.
- 대기 중인 유저는 자기가 **몇 번째인지 몰라** 계속 새로고침한다.
- 결제 지연/실패가 전체 요청을 막는다.

TICKETO는 이 "**순간 트래픽 + 유한 재고**" 문제를 정면으로 다루는 예매 서비스다. 제품으로서도 말이 되고, 기술적으로는 동시성·대기열·비동기 처리를 학습할 최적의 소재다.

## 2. 목표

**제품 목표**: 매진이 빠른 이벤트를 공정하고(선착순) 안정적으로(초과 판매 0) 예매할 수 있게 한다.

**학습 목표**: DB 설계·쿼리 최적화 / 동시성 제어 / 큐·비동기 / 인증·API 설계 / 배포·운영을 실제로 경험한다.

**비목표(Non-goals)**: 실제 결제(PG) 연동, 정산, 환불 정책 고도화, 모바일 앱. (결제는 **모의 결제 + 비동기 처리**로 대체 — 큐 학습은 그대로 가져가되 비용/복잡도는 줄인다.)

## 3. 타겟 사용자 & 시나리오

| 역할 | 하는 일 |
|---|---|
| 일반 유저 | 이벤트를 찾고, 오픈 시각에 대기열에 진입해, 순번이 오면 좌석/수량을 선택하고 결제한다. |
| 주최자(관리자) | 이벤트를 등록하고(좌석 수·오픈 시각·가격), 판매 현황을 대시보드로 본다. |

**핵심 시나리오**: "금요일 20:00 오픈. 60석짜리 재즈 공연에 300명이 동시 접속. 유저는 대기열에서 실시간 순번을 보며 기다리고, 입장하면 5분 안에 좌석을 잡아 결제한다. 시간 초과 시 좌석은 자동으로 다음 사람에게 넘어간다."

## 4. 핵심 기능 (MVP 범위)

1. **회원가입/로그인** (이메일 + JWT, 역할: user/admin)
2. **이벤트 목록·상세** (오픈 예정/판매중/매진 상태 표시)
3. **선착순 대기열** — 오픈 시각에 진입, **실시간 순번 표시**(SSE)
4. **좌석/수량 선택 + 홀드** — 5분 타임아웃, 초과 판매 방지
5. **모의 결제** — 비동기 처리(큐), 성공 시 예매 확정, 실패/타임아웃 시 좌석 반환
6. **내 예매 내역**
7. **관리자**: 이벤트 등록, 실시간 판매 현황 대시보드

**범위 밖(이번엔 안 함)**: 실 결제, 소셜 로그인, 이메일 실발송(로그로 대체), 좌석 배치도 UI(수량 기반으로 단순화).

## 5. 유저 플로우

```
[가입/로그인]
     │
     ▼
[이벤트 목록] ──> [이벤트 상세]
                     │ (오픈 전이면 카운트다운)
                     ▼ 오픈 시각 도달
                [대기열 진입] ──SSE──> "당신은 지금 152번째입니다"
                     │ 순번 도달
                     ▼
                [좌석/수량 선택] ── 홀드(5분 타이머) ──> [모의 결제]
                                                          │
                                       ┌──────────────────┼──────────────────┐
                                    성공                타임아웃/실패
                                       ▼                                     ▼
                                 [예매 확정]                          [좌석 반환 → 다음 대기자]
                                       ▼
                                 [내 예매 내역]
```

## 6. 화면 정의 (프론트엔드)

| 화면 | 주요 요소 | 학습 포인트(FE) |
|---|---|---|
| 로그인/가입 | 폼, 유효성 검사 | 인증 토큰 저장·인터셉터 |
| 이벤트 목록 | 카드 그리드, 상태 뱃지 | 서버 데이터 캐싱(TanStack Query) |
| 이벤트 상세 | 정보, 오픈 카운트다운, "대기열 입장" | 실시간 상태 전환 |
| **대기열** | 실시간 순번, 예상 대기시간, 진행바 | **SSE 구독**, 재연결 처리 |
| 좌석/수량 선택 | 수량 스텝퍼, 홀드 타이머(5분) | 만료 카운트다운, 낙관적 UI |
| 결제 | 모의 결제 버튼, 상태 폴링 | 비동기 상태(pending→done) 표현 |
| 내 예매 | 예매 리스트, 상태 | 페이지네이션 |
| 관리자 대시보드 | 이벤트 등록 폼, 판매 현황 차트 | 폼·차트, 권한 분기 |

**FE 스택**: Next.js(App Router) + TypeScript + TanStack Query + Tailwind. 배포는 Vercel(무료). → 공개 URL 포트폴리오로 바로 사용 가능.

## 7. 데이터 모델 (ERD)

```
┌──────────┐        ┌──────────────┐        ┌──────────────┐
│  User    │        │   Event      │        │  Inventory   │
├──────────┤        ├──────────────┤        ├──────────────┤
│ id (PK)  │        │ id (PK)      │ 1    1 │ id (PK)      │
│ email    │        │ title        │────────│ event_id(FK) │
│ password │        │ open_at      │        │ total_qty    │
│ role     │        │ price        │        │ remaining_qty│  ← 동시성 핵심 컬럼
│ created  │        │ status       │        │ version      │  ← 낙관적 락용
└────┬─────┘        └──────┬───────┘        └──────────────┘
     │ 1                   │ 1
     │                     │
     │ N                   │ N
┌────▼─────────────────────▼────┐        ┌──────────────┐
│        Reservation            │ 1    1 │  Payment     │
├───────────────────────────────┤────────├──────────────┤
│ id (PK)                       │        │ id (PK)      │
│ user_id (FK)                  │        │ reservation_ │
│ event_id (FK)                 │        │   id (FK)    │
│ quantity                      │        │ amount       │
│ status (HELD/CONFIRMED/       │        │ status       │
│         EXPIRED/CANCELLED)    │        │ idempotency_ │  ← 중복결제 방지
│ held_until                    │        │   key        │
│ created_at                    │        └──────────────┘
└───────────────────────────────┘
```

**관계 요약**: User 1—N Reservation, Event 1—N Reservation, Event 1—1 Inventory, Reservation 1—1 Payment.

**설계 의도**: 재고(`remaining_qty`)를 별도 Inventory로 분리해 락 경합 범위를 좁힌다. `version`으로 낙관적 락, `held_until`로 홀드 만료, `idempotency_key`로 중복 결제 방지 — 전부 학습 목표와 직결.

> 실제 개발 땐 Prisma 스키마로 정의하고, dbdiagram.io 등으로 시각적 ERD를 뽑아 README에 넣으면 포트폴리오 완성도가 올라간다.

## 8. 주요 API (초안)

| Method | Path | 설명 | 인증 |
|---|---|---|---|
| POST | /auth/signup, /auth/login | 가입/로그인 | - |
| GET | /events | 이벤트 목록 | - |
| GET | /events/:id | 상세 | - |
| POST | /events/:id/queue | 대기열 진입 | user |
| GET | /events/:id/queue/stream | 순번 SSE 스트림 | user |
| POST | /events/:id/reservations | 좌석 홀드(재고 차감) | user |
| POST | /reservations/:id/pay | 모의 결제(비동기 트리거) | user |
| GET | /reservations/me | 내 예매 내역 | user |
| POST | /admin/events | 이벤트 등록 | admin |
| GET | /admin/events/:id/stats | 판매 현황 | admin |

## 9. 성공 기준 (측정 가능)

- 동시 1,000+ 요청에서 **초과 판매 0건**.
- 재고 소진까지 **p95 응답 < 300ms**, 동시성 개선 전후 수치 비교 리포트.
- CI push → VM 자동 배포 파이프라인 동작.
- 공개 URL(FE: Vercel / BE: VM)로 실제 예매 데모 가능.

## 10. 일정 (4주 요약)

| 주차 | FE | BE / 인프라 |
|---|---|---|
| W0 | 와이어프레임, 프로젝트 셋업 | ERD·API 확정, docker-compose |
| W1 | 로그인·이벤트 목록/상세 | 인증(JWT), 이벤트 CRUD, 인덱스 |
| W2 | 대기열·좌석선택 UI | **동시성 실험(락 3종 + Redis) + k6** |
| W3 | 결제·내역·SSE 연동 | 대기열(Redis)·SSE·BullMQ 비동기 |
| W4 | 관리자 대시보드, 배포 | Docker·GitHub Actions·VM·Grafana |

> 상세 기술 로드맵과 스택 선정 이유는 별도 문서 `티켓팅_사이드프로젝트_기획.md` 참고.
