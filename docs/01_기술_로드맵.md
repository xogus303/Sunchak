# 선착순 티켓 예매 시스템 (Flash-Sale Ticketing) — 백엔드 심화 사이드 프로젝트

> 5년차 프론트엔드 → 풀스택 전환용. Node.js + TypeScript. 1개월 / 월 $15 미만.

---

## 왜 이 프로젝트인가

당신이 키우고 싶다고 한 세 가지 역량 **(1) DB 설계·쿼리 최적화, (2) 실시간/비동기/큐, (3) 인증/보안/API 설계** 를 "억지로 끼워넣지 않고 도메인이 자연스럽게 요구하는" 유일한 부류가 **선착순 예매**다.

- **순간 대량 트래픽 → 재고 일관성**: 좌석은 유한하고, 수천 명이 동시에 같은 좌석을 노린다. 초과 판매(oversell)를 막으려면 트랜잭션·락·원자적 연산을 진짜로 이해해야 한다. 이건 CRUD로는 절대 안 배워지는 부분이고, **면접에서 가장 자주 물어보는 "대용량 트래픽 처리 경험"** 그 자체다.
- **대기열 → 큐 + 실시간**: 입장 순번을 관리하고(Redis 대기열), 순번 변화를 실시간으로 밀어주고(SSE), 결제는 비동기 워커(BullMQ)로 처리한다.
- **인증/권한**: 로그인한 유저만 예매, 관리자만 이벤트 생성. JWT + RBAC의 정석 케이스.

주제는 "기술 학습 중심"이라 했지만, 티켓팅은 **토이가 아니라 기술 정당성이 명확한 도메인**이다. 포트폴리오로 써도 "동시성을 이렇게 풀었다"는 서사가 그대로 남는다.

## 기술 스택 (선정 이유 포함)

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | **NestJS** (TypeScript) | 5년차에 걸맞은 모듈/DI/레이어드 구조. Express보다 "백엔드 아키텍처"를 배운다. |
| DB | **PostgreSQL** | 트랜잭션 격리 수준·락을 제대로 실험할 수 있는 표준 RDB. |
| ORM | **Prisma** (일부 raw SQL 병행) | DX 좋고, 락/복잡 쿼리는 일부러 raw SQL로 내려가며 학습. |
| 캐시·원자연산 | **Redis** | 재고 원자 차감, 대기열, 캐싱, rate limit 전부 담당. |
| 큐 | **BullMQ** (Redis 기반) | 결제/알림 비동기 처리, 재시도·지연 작업 학습. |
| 실시간 | **SSE** (Server-Sent Events) | 대기열 순번 스트리밍엔 WebSocket보다 단순·충분. |
| 컨테이너 | **Docker / docker-compose** | 로컬은 app+postgres+redis 한 방에. |
| 배포 | **클라우드 VM + Nginx** | 서버리스로 도망가지 않고 "서버를 직접 운영"하는 감각. (추천: Oracle Cloud 무료 tier 또는 DigitalOcean $6 droplet) |
| CI/CD | **GitHub Actions** | build → test → 이미지 빌드(GHCR) → SSH 배포 파이프라인. |
| 관측 | **Prometheus + Grafana** | p95 레이턴시·에러율·큐 적체 대시보드. |
| 부하 테스트 | **k6** | 동시성 개선의 before/after를 숫자로 증명. |

> **인프라 깊이 결론**: "서버리스는 너무 얕고, K8s는 1개월에 오버킬." 그래서 **Docker + 클라우드 VM + CI/CD + 기본 관측**이 딱 현업 감각을 주는 스윗스팟이다. K8s/Terraform은 여유 있으면 마지막 주 스트레치로만.

## 아키텍처 개요

```
[Client] ──HTTP──> [Nginx] ──> [NestJS API]
                                  │  ├─ Auth (JWT/RBAC)
                                  │  ├─ Reservation (트랜잭션 + 락)
                                  │  └─ SSE (대기열 순번 스트림)
                                  │
                     ┌────────────┼─────────────┐
                     ▼            ▼              ▼
               [PostgreSQL]   [Redis]        [BullMQ Worker]
                (영속/재고)   (대기열/원자/캐시)  (결제/알림 비동기)
```

핵심 흐름:
1. 유저가 예매 요청 → **Redis 대기열**에 진입, 순번을 **SSE**로 실시간 수신.
2. 순번이 오면 좌석 **홀드(hold)** — Redis 원자 연산 또는 DB 락으로 재고 확보.
3. 결제 요청은 **BullMQ**로 비동기 처리, 타임아웃(예: 5분) 시 좌석 자동 해제.
4. 결제 성공 → DB 트랜잭션으로 예매 확정, 알림 발송(비동기).

## 4주 로드맵

### Week 0 (2~3일) · 설계
- 요구사항 정의, **ERD 설계** (User, Event, Seat/Inventory, Reservation, Payment).
- **API 설계** (REST, 리소스/상태 전이 명확히), 에러 규약, OpenAPI 문서.
- `docker-compose`로 postgres+redis 로컬 구동.
- **산출물**: ERD, API 명세서, 레포 스캐폴딩.

### Week 1 · 도메인 + DB + 인증
- 엔티티/마이그레이션, 이벤트·좌석 CRUD.
- **JWT 인증 + RBAC**(user/admin), 리프레시 토큰, 비밀번호 해싱(argon2).
- **인덱스 설계**, 조회 쿼리 최적화, N+1 잡기(쿼리 로깅으로 확인).
- **학습 포인트**: 트랜잭션 격리 수준, 인덱스가 실제로 먹는지 `EXPLAIN ANALYZE`로 확인.
- **산출물**: 로그인 + 이벤트 조회 API 동작.

### Week 2 · 동시성 (★ 이 프로젝트의 하이라이트)
- 먼저 **순진한 예매 구현**(조회 후 차감) → k6로 1,000 VU 동시 요청 → **초과 판매 버그 재현**.
- 순차적으로 해결하며 비교 실험:
  1. **비관적 락** (`SELECT ... FOR UPDATE`)
  2. **낙관적 락** (version 컬럼)
  3. **Redis 원자 차감** (`DECR` / Lua 스크립트)
- 각 방식의 **정합성 vs 처리량(throughput)** 을 표로 정리.
- **학습 포인트**: 데드락, 락 경합, 왜 티켓팅이 Redis를 쓰는가.
- **산출물**: "초과 판매 0건 @ 1,000 VU" + 방식별 p95 비교 리포트.

### Week 3 · 대기열 + 큐 + 실시간
- **Redis 대기열**(Sorted Set)로 입장 순서 관리, 처리량 제어(초당 N명 입장).
- **SSE**로 대기 순번 실시간 스트리밍.
- **BullMQ**로 결제/알림 비동기 처리 — 재시도, 지연 작업, **좌석 홀드 타임아웃 해제**.
- **멱등성(idempotency key)** 으로 중복 결제 방지.
- **학습 포인트**: 이벤트 기반 흐름, at-least-once 처리와 멱등성.
- **산출물**: 대기열 → 결제 → 확정 전 흐름 end-to-end 동작.

### Week 4 · 인프라 · 배포 · 관측
- **Dockerize**(멀티스테이지 빌드), 이미지 최적화.
- **GitHub Actions**: lint/test → 이미지 빌드 → GHCR push → VM에서 pull & 재기동.
- 클라우드 **VM 배포 + Nginx 리버스 프록시**(+ 무료 도메인/HTTPS는 옵션).
- **Prometheus + Grafana** 대시보드(요청률, p95, 에러율, 큐 적체, Redis hit).
- **k6 최종 부하 리포트** + README + 회고(트러블슈팅 기록).
- **공개 데모 모드**(ADR 0016) — 무료 티어 위에서 "누구나 안전하게 체험":
  - **진입 게이트**: 공유 데모 비번 → API 계층에서 데모 토큰 검사(신뢰 경계는 백엔드). 비번은 포트폴리오에 공개.
  - **데모 장치**: ① 서버측 부하 시뮬레이션(가상 유저 N명 투입 → 순번·재고 소진 실시간 재현) ② 실시간 판매 대시보드(SSE) ③ 데이터 리셋(자동 주기 + 수동 버튼).
  - **한도 보호**: 시뮬 버튼 상한(최대 VU) + 쿨다운(Redis rate limit).
- **산출물**: 공개 URL로 접속 가능한 배포본 + 대시보드 스크린샷 + **누구나 비번 입력 후 실시간 티케팅/순번을 체험 가능한 데모**.

## 성공 기준 (측정 가능하게)
- 동시 1,000+ 요청에서 **초과 판매 0건**.
- 재고 소진까지 **p95 레이턴시 목표치**(예: < 300ms) 달성 및 개선 전후 비교.
- CI push → 자동 배포까지 **무중단에 가깝게**(최소 재시작 다운타임 문서화).
- Grafana에서 부하 스파이크가 시각적으로 관측됨.
- **공개 데모에서 방문자가 비번 입력 → 시뮬레이션 → 실시간 순번·재고 소진을 눈으로 체험** 가능(무료 티어 한도 내, ADR 0016).

## 여유 있으면 (스트레치)
분산 락(Redlock), 읽기 복제(read replica), rate limiting 정교화, 블루-그린 배포, Terraform으로 VM 프로비저닝, K8s로 워커 스케일아웃.
