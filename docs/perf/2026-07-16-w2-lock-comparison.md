# W2 동시성 전략 비교 — 순진/비관/낙관/원자연산/Redis (k6 부하)

**작성:** 2026-07-16 · **관련:** DEVLOG(2026-07-16 W2), ADR 0009(재고 분리 설계)

선착순 예매의 재고 차감을 5가지로 구현하고, **같은 재고 행(hot row)에 요청이 몰릴 때**의
처리량·정확성을 k6로 계측했다. 목적은 "왜 이 전략을 고르는가"를 수치로 남기는 것.

## 측정 방법 (재현 정보)

| 항목 | 값 |
|---|---|
| 도구 | k6 v2.1.0 |
| 프리셋 | VU 30 · duration 15s · 각 전략마다 동일 |
| 재고 | 200,000 (도중 매진 방지 → 순수 "차감 처리 속도"를 측정) |
| 대상 | 단일 이벤트 1개 = 단일 재고 행에 전 VU가 경합(선착순 상황 모사) |
| DB | 로컬 Postgres 16 (docker-compose, `localhost:5432`) — Neon 아님(§9) |
| Redis | 로컬 Redis 7 (docker-compose, `localhost:6379`) — redis 전략 전용 재고 카운터 |
| 전환 | `POST /events/:id/reservations?strategy=<naive|pessimistic|optimistic|atomic|redis>` |
| 인증 | admin 토큰 1개 재사용(전 VU 공유) |
| 스크립트 | `apps/api/test/load/reservations_load.js`, `bench.sh` |

> ⚠️ 로컬·단일 실행 수치라 **절대값은 프로덕션 대표성이 없다.** 의미 있는 건 **전략 간 상대 비교**다.
> 커넥션 풀·머신 상태에 따라 회차별 편차가 있다.

## 결과

> 아래는 5전략을 한 번에 돌린 회차(2026-07-16). 전 회차 대비 절대값 편차는 있으나 상대 순위는 동일.

| 전략 | RPS | 성공(201) | 실패(409) | 실제 차감 | p95(ms) | 정확성 |
|---|---|---|---|---|---|---|
| naive | 2338 | 35,091 | 0 | **1,441** | 33.6 | ❌ 33,650건 lost update |
| pessimistic | 1126 | 16,912 | 0 | 16,912 | 67.9 | ✅ |
| optimistic | 859 | 12,902 | 8,262 | 4,640 | 61.9 | ⚠️ 정확하나 재시도 소진 대량 실패 |
| atomic | 2024 | 30,374 | 0 | 30,374 | 41.8 | ✅ |
| **redis** | **9354** | 140,329 | 0 | 140,329 | **4.4** | ✅ |

## 해석

- **naive** — RPS는 1위지만 예매 36,918건 중 실제 차감은 1,498건뿐. 나머지 **35,420건이 갱신 손실
  (lost update)**로 증발. 카운터엔 정상처럼 보이나 없는 좌석을 판 것. *속도는 정확성을 버린 대가.*
- **atomic** — RPS 2030(naive의 82%)으로 거의 최고 속도이면서 차감=예매=30,470으로 **완벽 일치**.
  단일 조건부 UPDATE라 DB가 행 락을 "그 순간만" 잡음 → 정확성과 처리량을 모두 확보. **이 시나리오의 승자.**
- **pessimistic** — 정확하지만 RPS는 atomic의 ~60%. 트랜잭션+`FOR UPDATE`로 행을 붙잡는 동안
  나머지가 대기(직렬화). p95 60ms로 지연도 최악. *정확성의 대가 = 대기 비용.*
- **optimistic** — 8,262건이 409로 실패(재고 20만이라 매진 아님 = **재시도 5회 소진**). 30 VU가 같은 행에
  version CAS를 던지니 대부분 경쟁에서 밀려 재시도→포기. 성공은 36%뿐. *경합 극심 시 thrashing으로 손해.*
- **redis** — RPS 9354로 **atomic의 4.6배**, p95 4.4ms로 지연도 최저이면서 차감=예매=140,329로 완벽 정확.
  경합하는 "재고 확인+차감"을 Postgres에서 Redis 인메모리(`DECRBY`)로 들어냈다. 그 결과:
  1. **직렬 구간의 1건당 비용이 급감** — Postgres의 hot row UPDATE엔 행 락·MVCC(새 행 버전)·WAL 기록이
     붙지만, Redis `DECRBY`는 RAM 정수 감산 하나뿐. 같은 "단일 카운터 직렬화"라도 Redis 줄이 훨씬 빨리 빈다.
  2. **DB에 남는 건 경합 없는 INSERT뿐** — hot row UPDATE가 사라져, Postgres는 서로 다른 행을 꽂는
     `reservation.create`를 커넥션 풀만큼 병렬 처리. 그래서 DB 쪽 처리량 상한도 함께 올라갔다.
  - 단, `DECRBY`엔 재고 부족 조건이 없어 0 밑으로도 깎인다 → **반환값이 음수면 `INCRBY`로 되돌리고 409**
    (앱이 매진을 판정·보상). *원자성은 lost update만 없애고, 경계 검사는 여전히 우리 몫.*

## 결론

```
정확성 없는 속도는 무의미                         → naive 탈락
hot row(=선착순) 고경합에서 처리량:
   redis  >>  atomic  >  pessimistic  >  optimistic
   RAM 차감    단일문장     락 대기        재시도 thrashing
```

- **선착순처럼 한 재고 행에 몰리는 상황 → 재고 차감을 Redis 인메모리로 옮기는 게 가장 빠르고 정확.**
  병목은 "DB가 느려서"가 아니라 **단일 재고 행 쓰기가 직렬화돼서**였고, 그 한 행을 Redis로 빼면 풀린다.
- **비용: 정합성.** 재고의 진짜 값이 Redis에만 있어 DB `inventories`와 어긋나고, Redis 유실 시 재구성이 필요.
  실서비스는 Redis로 선착순 관문만 통과시키고 DB 반영은 뒤에서 맞춘다(→ 큐/BullMQ의 역할).
- **DB만으로 간다면 원자연산(조건부 단일 UPDATE)이 차선**: 인프라 추가 없이 빠르고 정확·단순.
- **낙관적 락은 경합이 드문 경우에나 유리**, **비관적 락은 확실한 직렬화가 필요할 때의 안전판.**
