# 0008. 프론트엔드: Next.js

- 상태: Accepted
- 날짜: 2026-07-10

## 맥락 (Context)
결과물을 포트폴리오로 쓰려면 공개 URL로 배포된 실제 화면이 필요하다. 개발자는 5년차 프론트엔드라 React 역량이 이미 있고, 초점은 백엔드 학습이므로 프론트는 "빠르게 좋은 결과물"이 목표다.

## 결정 (Decision)
프론트엔드는 **Next.js(App Router) + TypeScript**, 배포는 Vercel(무료).

## 고려한 대안 (Alternatives)
| 대안 | 장점 | 단점 / 채택하지 않은 이유 |
|---|---|---|
| **Vite + React SPA** | 가볍고 빠름, 순수 CSR엔 충분. | 배포·라우팅·메타 등 부수 작업을 직접 챙겨야 함. 포트폴리오용 폴리시엔 Next의 기본기가 더 나음. |
| **Remix** | 웹 표준·데이터 로딩 모델이 우수. | 사용자의 기존 자산·시장 친숙도 면에서 Next가 우위. 학습 초점이 백엔드라 프론트에서 새 패러다임 비용을 지출할 이유가 적음. |
| **Next.js** | React 생태계 표준, Vercel 원클릭 배포로 공개 URL 확보가 즉시. SSR/CSR 유연, 포트폴리오 완성도 높음. | 백엔드 학습이 목적이라 Next의 서버 기능(API routes 등)은 최소로만 사용(백엔드는 NestJS가 담당). |

## 근거 (Rationale)
프론트는 이미 강점 영역이므로 익숙하고 배포가 즉시 되는 Next.js로 시간을 아끼고, 아낀 시간을 백엔드 학습에 쓴다. Vercel 무료 배포로 "공개된 실제 서비스"라는 포트폴리오 요건을 손쉽게 만족한다. Next의 서버 기능은 의도적으로 최소화하고, 서버 로직은 NestJS로 몰아 백엔드 학습 목적을 흐리지 않는다.

## 결과 (Consequences)
- 공개 URL 확보가 쉬워 포트폴리오 요건 충족.
- 프론트/백엔드 경계를 명확히(데이터는 NestJS API에서) 유지하는 규율이 필요.
