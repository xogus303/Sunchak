# 0015. 선착순 예매 정합성 설계 — HELD 선(先)기록 + BullMQ 큐 + 멱등성 + SSE

- 상태: Accepted (2026-07-20 멱등성 부분 개정)
- 날짜: 2026-07-19
- 관련: 0004(Redis), 0005(BullMQ), 0006(SSE), 0009(스키마), 0014(예매 전략)

> **개정 이력 (2026-07-20 — 구현 착수 직전 검토에서 발견)**
> 최초 작성본은 멱등성 키를 **API 서버가 발급**하고, 그 unique 제약이 **워커 재시도의 중복 INSERT**를 막는다고 썼다. 둘 다 틀렸다.
> - 이 파이프라인에서 INSERT는 요청 경로(②)에서 **한 번만** 일어나고, 워커(⑤)는 UPDATE만 한다. 재시도돼도 "두 번째 INSERT"는 발생하지 않는다 → unique가 막을 대상이 없었다.
> - 서버가 키를 발급하면 요청마다 새 키가 나오므로 "같은 요청"을 정의할 수 없다. 그건 멱등성 키가 아니라 PK다.
> 아래 본문은 **클라이언트 발급 키 + 관문 보상(`INCRBY`)** 으로 정정된 내용이다. 워커 재시도용 별도 장치는 필요 없다(UPDATE가 본래 멱등).

## 맥락 (Context)

ADR 0014에서 재고 차감을 **Redis 인메모리 관문(`DECRBY`)**으로 처리하기로 했다. 하지만 관문만으로는 세 가지 정합성 문제가 남는다:

1. **Redis ↔ DB 어긋남**: 관문에서 재고는 깎였는데(Redis) 실제 예매 기록(DB) INSERT가 실패하면, 좌석 하나가 증발한다.
2. **Redis 유실 재구성**: Redis는 휘발성이라 죽으면 재고 카운터가 날아간다. 어떻게 정확히 복구하나?
3. **멱등성(중복)**: 예매는 성공했는데 응답이 유실돼 **클라이언트가 같은 요청을 재전송**하면(또는 사용자가 두 번 누르면), 같은 예매가 두 번 기록될 수 있다. 서버는 이 재전송과 "정말 한 장 더 사려는 두 번째 주문"을 구분할 수 없다.

즉 "관문 통과 후, 실제 예매 기록을 어떻게 신뢰성 있게 남기고 · 사용자에게 확정을 알리고 · Redis 유실에 대비하는가"를 설계해야 한다.

## 결정 (Decision)

관문 통과 후를 다음 파이프라인으로 처리한다.

```
[요청]  ← 클라이언트가 멱등성 키(UUID)를 실어 보냄
  ① 관문       Redis DECRBY  (초과판매 차단 — hot row 직렬화를 인메모리로)
  ② HELD 선기록 DB에 status=HELD INSERT  (경합 없는 병렬 INSERT)
       └ unique 위반(P2002) = 재전송 → INCRBY 보상 + 기존 예매를 그대로 응답(끝)
  ③ 큐잉       BullMQ에 job 투입
  ④ 즉시 응답   "접수됨(HELD)"   ← 사용자 응답과 DB 확정을 분리(decoupling)
       ┊
  [워커: 별도 프로세스]
  ⑤ HELD → CONFIRMED  UPDATE  (WHERE status=HELD — 재실행 시 0건이라 본래 멱등)
  ⑥ 실패 시 재시도  (at-least-once — ⑤가 멱등이라 중복 확정 없음)
  ⑦ SSE로 "확정" push
       ┊
  [안전장치]
  · HELD 만료(TTL): 결제 미완/붕 뜬 자리 자동 회수
  · Redis 유실 재구성: remaining = 총재고 − (HELD + CONFIRMED)   ← DB만으로 정확 복구
```

- **멱등성 키는 클라이언트가 발급**한다(요청당 UUID 1개, 재전송 시 **같은 키 유지**). "이 두 요청이 같은 의도인가"는 재전송을 실행한 쪽만 알 수 있다. 서버가 발급하면 매 요청 새 키가 나와 중복을 정의할 수 없다.
- **중복 방어**: `reservation`에 멱등성 키 컬럼 + **unique 제약**. 재전송의 INSERT를 DB가 원자적으로 거부한다(앱에서 `SELECT` 후 `INSERT`하면 그 틈에 중복이 끼어든다 — 아래 대안 표).
- **보상(compensation)**: 관문(①)이 INSERT(②)보다 먼저라, 재전송도 `DECRBY`를 한 번 더 깎고 나서야 unique에 걸린다. 그대로 두면 **주문 없이 재고만 증발**한다. 따라서 unique 위반 시 **`INCRBY`로 되돌린다**(Redis 전략에서 재고 음수 시 쓰던 보상과 같은 동작).
- **재전송의 응답은 성공**이다. 이 사용자는 실패한 게 아니라 이미 성공했고 응답만 못 받았다. 409가 아니라 **첫 요청과 동일한 결과(같은 예매 ID)** 를 반환한다 — "몇 번을 호출하든 결과가 같다"가 멱등성의 정의다.
- **워커 재시도엔 별도 장치가 불필요**하다. ⑤는 `WHERE status=HELD` UPDATE라 두 번째 실행은 0건으로 끝난다(본래 멱등).
- **상태 전이**: `HELD`(관문 통과, 확정 대기) → `CONFIRMED`(워커가 DB 확정) → (미확정 시)`EXPIRED`(TTL 회수).

## 고려한 대안 (Alternatives)

| 주제 | 대안 | 채택하지 않은 이유 |
|---|---|---|
| 확정 통보 | **폴링**(클라이언트가 주기적 GET) | 수만 명 동시 대기 시 요청 폭탄 + 확정 시점 지연. 서버가 push하는 **SSE**(0006)가 실시간·효율적. 단방향이라 WebSocket보다 가벼움. |
| HELD 위치 | Redis/큐에만 두고 DB엔 CONFIRMED만 | Redis 유실 시 DB에 흔적이 없어 재구성이 부정확(HELD를 셈할 수 없음). 큐도 Redis 위에 살아 함께 유실. → **DB에 HELD 선기록**해야 재구성·복구 가능. |
| 중복 방지 | 앱에서 `SELECT(있나?) → INSERT` | SELECT와 INSERT 사이 틈에 중복 삽입(W2 초과판매와 같은 read-check-write 함정). **unique 제약**이 원자적으로 차단. |
| 기록 방식 | 관문 직후 **동기** DB INSERT | 사용자가 DB 완료까지 대기(지연) + 실패 시 유실. **큐 비동기**가 빠른 응답 + 재시도 신뢰성. |

## 근거 (Rationale)

- **왜 HELD를 DB에 먼저 남기나**: HELD는 "관문은 통과했지만 아직 확정 전"인 미확정 주문이다. 이게 Redis/큐에만 있으면 Redis 유실 시 DB로 재구성할 때 셈할 수 없어 재고가 부정확해진다. 최초 시점부터 DB에 흔적을 남겨야 `총재고 − (HELD+CONFIRMED)`로 정확히 복구된다.
- **HELD 기록은 hot row가 아니다**: 재고 차감(①)은 단일 재고 행 경합이라 Redis로 뺐지만, HELD INSERT(②)는 **서로 다른 주문 행**이라 경합 없는 병렬 작업이다. 원래 병목(단일 행 직렬화)은 관문이 이미 해결했으므로 요청 경로에 INSERT가 남아도 부담이 작다. **직렬화 병목 ≠ INSERT 비용**(W2의 핵심 교훈).
- **각 단계의 "원자성"은 각기 다른 문제를 푼다**: ① 관문 `DECRBY` = **초과판매**(서로 다른 사용자들이 같은 재고를 다툼) / ② unique 제약 = **중복**(같은 사용자의 같은 요청이 두 번 도착) / ⑤ 워커 UPDATE = 확정(별도 장치 없이 그 자체로 멱등). 같은 "원자"라도 대상·목적이 다르다.
- **막을 수 있는 중복과 없는 중복**: 서버는 "재전송"과 "진짜 두 번째 주문"을 요청만 보고 구분할 수 없다. 그래서 판정 기준(멱등성 키)을 **클라이언트에서 받아온다**. 클라이언트가 키를 매번 새로 만들어 보내면 두 번째 주문으로 처리되는데, 이는 설계 실패가 아니라 **판정 권한을 클라이언트에 위임한 결과**다.
- **보상 vs 재구성은 서로 대체하지 않는다**: `INCRBY` 보상은 **즉시** 정확히 되돌리는 1차 방어고, Redis 재구성(`총재고−(HELD+CONFIRMED)`)은 **주기적** 사후 안전망이다. 보상을 빼도 재구성이 언젠가 고치지만, 그 사이 재고가 남았는데 관문이 막는 **거짓 품절**이 발생한다. 선착순에서는 바로 그 몇 분이 승부처다.
- **큐의 두 가치**: 비동기 분리(빠른 응답) + 신뢰성(job 영속·재시도로 유실 방지 = 최종 일관성).

## 결과 (Consequences)

**이득**
- 관문 통과 즉시 응답(빠름) + 워커 재시도로 결국 반드시 기록(최종 일관성) + Redis 유실에도 DB로 정확 재구성 + 확정을 실시간 통보.

**감수할 것**
- **요청 경로에 HELD INSERT가 남음**(단, 병렬이라 경미).
- **최종 일관성 지연**: "접수됨"과 "확정" 사이 시차 존재 → 그래서 SSE 통보가 필요.
- **운영 복잡도 증가**: 워커 프로세스, 큐(BullMQ), SSE 연결 관리, TTL 회수 잡.
- **Redis ↔ DB 이원화**를 코드가 계속 관리(재구성·멱등·만료).
- **멱등성의 책임 일부가 클라이언트로 넘어감**: 웹(apps/web)이 키를 생성하고 **재전송 시 유지**해야 한다. 클라이언트가 규약을 어기면 서버는 중복을 막을 수 없다.

**후속 작업(구현 시)**
- **스키마 변경**: `reservation`의 `status`(HELD/CONFIRMED/EXPIRED)·`heldUntil`(TTL 판정)·`@@index([status, heldUntil])`는 **W1 스키마(0009)에 이미 존재**한다. 실제로 추가할 것은 **멱등성 키 컬럼 + `@unique`** 하나뿐이다.
- BullMQ 큐/워커 모듈, SSE 엔드포인트, HELD 선기록 + 워커 확정 로직, TTL 만료·Redis 재구성 잡.
- 구현은 다음 세션(개념은 본 ADR로 확정). 세부는 구현 중 조정될 수 있으며, 크게 바뀌면 후속 ADR로 남긴다.
