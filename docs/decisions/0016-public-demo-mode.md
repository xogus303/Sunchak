# 0016. 공개 데모 모드 — 진입 게이트 + 데모 장치 + 데이터 리셋

- 상태: Accepted
- 날짜: 2026-07-21
- 관련: 0006(SSE), 0010(Neon), 0011(Infisical), 0013(인증), 0014·0015(예매)

## 맥락 (Context)

최종 산출물은 **공개 URL로 접속 가능한 포트폴리오 데모**다(로드맵 W4, 기획안 §9). 그런데 "그냥 배포"만으로는 두 가지가 성립하지 않는다.

1. **핵심 가치가 눈에 안 보인다.** 이 프로젝트의 무기는 "순간 대량 트래픽 → 초과판매 0 + 실시간 순번"이다. 그런데 방문자가 **혼자** 브라우저에서 예매 버튼을 한 번 누르면, 겉보기엔 평범한 CRUD와 구별되지 않는다. 동시성·대기열·재고 소진이 **재현되지 않으면** 포트폴리오로서 가장 강한 서사가 통째로 안 보인다.

2. **공개 URL이 무료 티어 위에 있다.** DB는 Neon 무료 tier(0010), 백엔드는 무료/저가 VM이다. 공개되면 크롤러·봇·장난 트래픽이 붙고, 특히 아래에서 추가할 **부하 시뮬레이션 버튼**을 아무나 남발하면 무료 한도를 태운다. 즉 **누구나 들어오되, 아무나 자원을 무한정 소모하진 못하게** 해야 한다.

이 ADR은 "공개 데모를 어떻게 **보여줄 만하게** 만들고, 동시에 **한도 안에서 안전하게** 지키는가"를 정한다.

## 결정 (Decision)

데모 모드를 세 축으로 구성한다.

### A. 진입 게이트 (공유 비번 → API 계층에서 검사)

- 방문자는 랜딩에서 **공유 데모 비번**(env `DEMO_GATE_PASSWORD`)을 입력한다. 통과하면 서버가 **단기 데모 토큰**을 발급하고, 이후 모든 API 요청이 이 토큰을 요구한다.
- **게이트는 반드시 API(백엔드) 계층에 둔다.** 프론트(Vercel)에만 비번을 걸면 봇이 프론트를 우회해 VM의 API를 직접 때릴 수 있다. 무료 티어 부하는 **백엔드/DB에서** 발생하므로, 방어선(신뢰 경계, trust boundary)은 API에 있어야 한다. 프론트 게이트 화면은 **UX 보조막**일 뿐 진짜 방어선이 아니다.
- 비번은 **포트폴리오/이력서에 함께 공개**한다(예: "데모 비번 `sunchak`"). 목적은 비밀 유지가 아니라 **크롤러·자동 봇의 원천 차단** + 방문자에게 "여기는 데모"라는 맥락 부여다.

> **게이트 ≠ 로그인.** 이 프로젝트엔 이미 JWT 로그인(user/admin, 0013)이 있다. 게이트는 그것과 **층위가 다른 별개의 막**이다. 로그인은 "예매 주체가 누구인가"를 식별하고, 게이트는 "이 데모에 들어올 자격이 있나"로 **서비스 전체 진입을 차단**한다. 목적·대상이 다르므로 별도 막으로 둔다.

### B. 데모 장치 3종 (핵심 가치를 눈에 보이게)

1. **부하 시뮬레이션 — 서버측 가상 유저.** 방문자가 "가상 유저 N명 투입" 버튼을 누르면, **서버가** N개의 예매 요청을 스스로 생성해 파이프라인(관문 → HELD → 큐 → 확정)에 흘린다. 그러면 대기열이 채워지고 순번이 실시간으로 줄고 재고가 소진되는 과정이 방문자 눈앞에서 재현된다.
2. **실시간 판매 대시보드 — SSE push.** 재고 잔량·HELD/CONFIRMED 카운트·큐 적체를 **SSE(0006)로 실시간 스트리밍**한다. 시뮬레이션이 도는 동안 숫자가 움직이는 이 화면이 데모의 하이라이트다.
3. **데이터 리셋 (아래 C).**

### C. 데이터 리셋 (자동 주기 + 수동 버튼, 둘 다)

재고가 한 번 소진되면 다음 방문자는 "매진"만 본다 → 재현 불가. 그래서 초기화 장치를 **둘 다** 둔다.

- **자동 주기 리셋**: 스케줄러가 일정 주기(env `DEMO_RESET_INTERVAL_MS`, 예: 10분)마다 데모 이벤트의 재고·예매·Redis 카운터를 초기화하고 `open_at`을 재설정한다. → 방문자는 언제 와도 "오픈 예정/판매중"을 본다(**상시성**).
- **수동 리셋 버튼**: 방문자가 지금 즉시 초기화. → 자동 주기를 기다리지 않고 바로 재현(**즉시성**).
- 리셋은 시드 로직을 재사용한다(STATUS에 적힌 seed 미비를 이 작업에서 함께 해소).

### D. 한도 보호 (시뮬 상한 + 쿨다운)

게이트를 통과한 사람도 시뮬 버튼을 남발하면 무료 티어를 태운다. 그래서 **시뮬레이션 버튼에만** 상한을 건다.

- **1회 최대 가상 유저 수** 제한(env `DEMO_SIM_MAX_VU`).
- **재실행 쿨다운**(env `DEMO_SIM_COOLDOWN_MS`) — Redis로 마지막 실행 시각을 기록해 쿨다운 내 재요청을 거부(429). Redis rate limit은 스택에 이미 있는 도구다.

## 고려한 대안 (Alternatives)

| 주제 | 대안 | 채택하지 않은 이유 |
|---|---|---|
| 게이트 위치 | 프론트(Vercel 비번 보호)만 | 봇이 프론트를 우회해 VM API를 직접 호출 가능. 무료 티어 부하는 백엔드에서 나므로 방어선이 뚫린다. → **API 계층 게이트**. |
| 게이트 방식 | 데모 전용 로그인 계정 하나 배포 | 로그인은 "신원 식별"이지 "전체 진입 차단"이 아니다(층위 혼동). 미로그인 공개 엔드포인트(이벤트 목록 등)는 그대로 노출됨. → **별도 진입 게이트**. |
| 봇 억제 | rate limit만(비번 없이) | 크롤러가 계속 두드리며 한도를 갉음. 비번 게이트가 **원천 차단** + 통과자 남발은 상한으로. → **게이트 + 시뮬 상한** 조합. |
| 시뮬 위치 | 방문자 브라우저에서 다중 요청 발사 | 브라우저는 진짜 동시성 재현이 약하고 방문자 회선·기기에 의존해 **비결정적**. 서버측이 결정적이고 무료 티어에서 **통제 가능**. → **서버측 가상 유저**. |
| 리셋 전략 | 자동만 / 수동만 / 방문자별 격리 | 자동만=즉시 재현 불가, 수동만=상시성 없음. 방문자별 격리 이벤트는 세션별 데이터 분리 설계라 복잡도 과함(단일 사용처에 오버엔지니어링). → **자동+수동 둘 다**. |

## 근거 (Rationale)

- **신뢰 경계를 백엔드에 둔다**는 게 이 설계의 학습 핵심이다. "방어를 어디에 치느냐"는 백엔드에서 반복해서 나오는 판단이고, "부하가 실제로 발생하는 곳(DB/서버)을 기준으로 방어선을 잡는다"가 정답이다. 프론트 게이트는 편의일 뿐 경계가 아니다.
- **게이트와 로그인의 분리**는 "인증(누구냐) vs 인가·진입(들어올 수 있냐)"의 층위 구분을 명확히 하는 사례다. 하나로 합치면 목적이 흐려지고 공개 엔드포인트에 구멍이 남는다.
- **데모 장치는 서사를 시각화하는 장치**다. W2에서 만든 "초과판매 0", W3의 "실시간 순번"은 숫자·코드로만 있으면 면접관·방문자에게 안 보인다. 시뮬 + 실시간 대시보드가 그 서사를 **눈에 보이는 데모**로 바꾼다 — 포트폴리오 가치와 직결.
- **자동+수동 리셋**은 상충하는 두 요구(항상 켜져 있는 상시 데모 vs 지금 당장 재현)를 각각의 장치로 동시에 만족시킨다. 하나를 고르면 반대쪽을 잃는다.
- **SSE 재사용**: 데모 대시보드의 실시간 스트림은 W3 2.4에서 만들 SSE와 같은 메커니즘이다. 2.4 SSE를 설계할 때 "확정 push"뿐 아니라 "stats 스트림"도 염두에 두면 이 데모 대시보드가 자연스럽게 얹힌다.

## 결과 (Consequences)

**이득**
- 무료 티어를 태우지 않으면서 **누구나 안전하게 접속**하는 공개 데모.
- 동시성·대기열·재고 소진이 방문자 눈앞에서 재현되는 **"보여줄 만한" 데모**.
- seed/리셋 로직 정비(STATUS의 알려진 틈 해소).

**감수할 것**
- **데모 전용 코드 경로**가 운영 로직에 얹힌다(게이트 가드, 시뮬 엔드포인트, 리셋 잡). 실제 프로덕션엔 없을 것들이므로 **데모 전용임을 코드·문서에 명시**하고 격리한다.
- **자동 리셋이 데이터를 주기적으로 날린다** — 데모라 의도된 동작이지만, 방문자가 만든 예매도 함께 사라짐을 UI에 안내한다.
- 게이트·시뮬 상한·리셋 잡의 유지보수 비용.
- **새 env 키 4개**: `DEMO_GATE_PASSWORD`, `DEMO_SIM_MAX_VU`, `DEMO_SIM_COOLDOWN_MS`, `DEMO_RESET_INTERVAL_MS`. `.env.example`에 값 없이 반영(§4), 비밀값(비번)은 Infisical(0011)에.

**후속 작업(구현 시)**
- **구현 시점은 W4**(배포·관측과 함께). 단 **W3 2.4 SSE 설계 때 stats 스트림을 염두**에 둔다(위 근거).
- 게이트 가드 + 데모 토큰 발급, 서버측 시뮬 엔드포인트(+상한·쿨다운), 실시간 stats SSE, 자동 리셋 스케줄러 + 수동 리셋 엔드포인트, 프론트 게이트/데모 컨트롤 화면.
- 세부는 구현 중 조정될 수 있으며, 크게 바뀌면 후속 ADR로 남긴다.
