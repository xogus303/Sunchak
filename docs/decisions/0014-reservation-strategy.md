# 0014. 선착순 예매 재고 차감 전략: Redis 인메모리 관문 + DB 비동기 기록

- 상태: Accepted
- 날짜: 2026-07-18
- 관련: 0003(PostgreSQL), 0004(Redis), 0005(BullMQ), 0009(재고 분리 스키마), W2 성능 비교(`docs/perf/2026-07-16-w2-lock-comparison.md`)

## 맥락 (Context)

선착순 티켓 예매의 본질은 **하나의 재고 숫자에 다수 요청이 동시에 몰리는 것**이다. 이 재고는 구조적으로 쪼갤 수 없다 — "이 이벤트의 남은 좌석"은 개념상 단일 값이라, 여러 행으로 분산해 병렬화하는 회피가 불가능하다. 즉 **hot row(단일 재고 행) 경합은 피할 수 없는 전제**다.

이 전제 위에서 재고 차감을 어떻게 처리하느냐에 따라 (1) **정확성**(초과판매 없음)과 (2) **처리량**(선착순 몰림을 버티는 RPS)이 갈린다. W2에서 5가지 전략을 구현하고 k6로 계측해 근거를 확보했다(같은 hot row에 VU 30 경합).

## 결정 (Decision)

**재고 차감(선착순 관문)은 Redis 인메모리 원자 연산(`DECRBY`)으로 처리하고, 실제 예매 기록의 DB 반영은 뒤에서(비동기 큐) 맞춘다.**

- **관문(gate)**: Redis에 재고 카운터를 두고 `DECRBY`로 차감. 반환값이 음수면 `INCRBY`로 보상하고 409(매진). → 선착순 당락을 Redis가 즉시 판정.
- **기록(record)**: 관문을 통과한 예매만 DB에 남긴다. hot row UPDATE가 사라졌으므로 DB에는 **서로 다른 행을 꽂는 경합 없는 INSERT**만 남아 병렬 처리된다.
- **정합성 보정**: Redis(진실의 원천, 재고)와 DB(예매 기록) 사이 어긋남·유실·중복은 큐(BullMQ)와 보정 로직으로 맞춘다. → 상세 설계는 W3에서 후속 ADR.
- **DB 단독 폴백**: Redis를 도입하지 않는 경로에서는 **원자연산(조건부 단일 `UPDATE ... WHERE remainingQty >= n`)**을 차선으로 둔다.

## 왜 Redis가 빠른가 — 핵심 논리 (Rationale)

병목은 "DB가 느려서"가 아니다. **hot row 쓰기가 직렬화되기 때문**이다. 그리고 Redis도 단일 스레드라 똑같이 직렬화된다. 차이는 **직렬 1건당 비용**에서 갈린다.

1. **hot row는 못 피한다.** 재고는 단일 숫자라 락 없이 병렬화할 방법이 없다. 재고 행을 20만 개로 흩어도 이 이벤트의 재고는 여전히 하나다. → DB든 Redis든 그 한 지점은 직렬 처리된다.

2. **DB의 직렬 1건당 비용이 크다.** Postgres가 hot row를 `UPDATE`할 때마다:
   - **행 락(row lock)**: 같은 행을 쓰려는 다른 트랜잭션을 커밋까지 대기시킴 → 격리(Isolation) 보장.
   - **MVCC 새 행 버전 생성**: 기존 행을 덮지 않고 새 버전 tuple을 만들고 트랜잭션 도장을 찍음. 덕에 읽기가 안 막히지만, UPDATE가 "복사본 생성" 비용이 됨. → 동시성 + 격리.
   - **WAL fsync**: 커밋 시 변경 로그를 **물리 디스크에 동기로** 강제 기록(fsync) → 지속성(Durability) 보장. 메모리 대비 수십~수백 배 느린 이 디스크 왕복이 최종 병목.

   즉 DB가 무거운 건 낭비가 아니라 **ACID(특히 Isolation·Durability)를 보장하려는 대가**다.

3. **Redis는 그 보장들을 (일부) 포기해 비용을 없앤다.**
   - 락 없음(단일 스레드라 애초에 동시 접근 없음), MVCC 없음(메모리 정수 in-place 감산).
   - fsync를 **요청 경로 밖으로** 뺀다(AOF `everysec` = 초당 1회 백그라운드). DB가 커밋마다 동기로 기다리는 fsync를, Redis는 비동기로 미룬다. → 요청은 디스크를 안 기다리고 메모리 속도로 응답.

**결과(수치, W2)**: redis RPS 9,354 / p95 4.4ms / 차감=예매 완벽 일치 — atomic의 **4.6배** 처리량. (naive는 RPS 최고지만 lost update로 초과판매, optimistic은 고경합 재시도 thrashing, pessimistic은 락 대기로 최저 처리량.)

## 고려한 대안 (Alternatives)

같은 hot row에 VU 30 경합(재고 20만, 매진 회피 = 순수 차감 속도) 계측:

| 전략 | 처리량(RPS) | 정확성 | 채택하지 않은 이유 |
|---|---|---|---|
| **naive** (읽기→확인→덮어쓰기) | 2,338 (최고) | ❌ lost update로 초과판매 | 빠르지만 없는 좌석을 판다. 정확성 없는 속도는 무의미. |
| **pessimistic** (`FOR UPDATE`) | 1,126 (최저) | ✅ | 정확하나 행을 붙잡는 동안 나머지가 대기(직렬화). p95 최악. |
| **optimistic** (version CAS + 재시도) | 859 | ⚠️ 정확하나 대량 실패 | 고경합에서 대부분 CAS 경쟁에 밀려 재시도 소진(thrashing). 성공 36%뿐. |
| **atomic** (조건부 단일 UPDATE) | 2,024 | ✅ | 정확·단순·빠름. **인프라 없이 DB만으로 갈 때의 차선**으로 채택(폴백). 단 hot row 건당 비용(락·MVCC·WAL)은 여전. |
| **redis** (인메모리 `DECRBY`+보상) | **9,354** | ✅ | **채택.** 직렬 1건당 비용을 메모리 수준으로 낮춰 처리량·지연 모두 최고. |

## 결과 (Consequences)

**이득**
- 선착순 몰림에서 최고 처리량·최저 지연으로 초과판매 없이 관문을 통과시킨다.
- DB에서 hot row UPDATE가 사라져 DB는 경합 없는 병렬 INSERT만 담당 → DB 처리량 상한도 함께 상승.

**감수할 것 (대가 = 정합성·지속성)**
- **진실의 원천 이원화**: 재고의 실시간 값은 Redis에, 예매 기록은 DB에 → 둘이 어긋날 수 있음.
- **유실 위험**: fsync를 미룬 만큼(최대 ~1초) 크래시 시 최근 차감이 날아갈 수 있음. Redis 재시작 시 카운터 재구성 필요.
- **경계 검사는 여전히 우리 몫**: `DECRBY`엔 "0 밑으로 못 내려감" 조건이 없다 → 음수 반환 시 `INCRBY` 보상 + 409로 앱이 매진을 판정.
- **중복 방지 필요**: 같은 요청 재시도 시 이중 차감 방지(멱등성).

→ 이 대가들(Redis↔DB 정합성, 유실 재구성, 중복 방지)을 큐(BullMQ) 기반 비동기 반영으로 어떻게 맞출지는 **W3에서 후속 ADR**로 확정한다.
