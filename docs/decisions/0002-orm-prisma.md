# 0002. ORM: Prisma

- 상태: Accepted
- 날짜: 2026-07-10
- 관련: 0003(PostgreSQL)

## 맥락 (Context)
DB 설계·쿼리 최적화가 핵심 학습 목표 중 하나다. 스키마를 코드로 관리(마이그레이션)하고, 동시성 실험(W2)에서는 락(`SELECT ... FOR UPDATE`) 같은 저수준 쿼리까지 내려가야 한다.

## 결정 (Decision)
ORM으로 **Prisma**를 쓰되, 락/복잡 쿼리는 **raw SQL로 의도적으로 내려간다**.

## 고려한 대안 (Alternatives)
| 대안 | 장점 | 단점 / 채택하지 않은 이유 |
|---|---|---|
| **TypeORM** | NestJS와 통합 예제가 많고 데코레이터 기반 엔티티가 Nest와 결이 같다. 리포지토리 패턴, 비관/낙관 락 API 제공. | 타입 안정성·마이그레이션 신뢰성 이슈가 종종 지적됨. API가 방대하고 함정이 있음. |
| **Drizzle** | SQL에 가깝고 가벼우며 타입이 강력. 최신 트렌드. | 상대적으로 신생이라 자료가 적어 학습 마찰이 큼. |
| **Knex (쿼리 빌더)** | SQL을 거의 그대로 다뤄 쿼리 감각을 가장 많이 배움. | 매핑/마이그레이션을 많이 수작업. 생산성이 낮아 1개월 일정에 부담. |
| **Prisma** | 최고의 타입 안전성·DX, 명료한 마이그레이션, 스키마 파일로 ERD가 곧 코드. `$queryRaw`로 raw SQL 병행 가능. | 기본 API로는 락 같은 저수준 제어가 제한적(그래서 raw SQL 병행). |

## 근거 (Rationale)
Prisma의 `schema.prisma`는 그 자체가 사람이 읽는 ERD라 문서화·학습에 좋다. 타입 안전성이 높아 실수를 줄이고 개발 속도를 확보한다. 동시에, 동시성 실험에서 일부러 `$queryRaw`로 `FOR UPDATE`·원자 UPDATE를 직접 써 보며 "ORM이 감춰주는 것"과 "직접 SQL로 제어하는 것"의 차이를 체감한다 — 이 대비 자체가 학습 포인트다.

## 결과 (Consequences)
- 스키마→마이그레이션→타입 생성 흐름이 매끄러워 W1이 빨라진다.
- W2에서 Prisma 기본 API의 한계를 만나 raw SQL로 내려가는 경험을 의도적으로 확보한다(락 3종 비교의 재료).
