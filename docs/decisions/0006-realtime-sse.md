# 0006. 실시간 전송: SSE (Server-Sent Events)

- 상태: Accepted
- 날짜: 2026-07-10

## 맥락 (Context)
대기열에서 유저에게 "당신은 지금 152번째"를 실시간으로 밀어줘야 한다. 방향은 서버 → 클라이언트 단방향이고, 데이터는 짧은 순번 갱신이다.

## 결정 (Decision)
실시간 순번 전송에 **SSE(Server-Sent Events)** 를 사용한다.

## 고려한 대안 (Alternatives)
| 대안 | 장점 | 단점 / 채택하지 않은 이유 |
|---|---|---|
| **폴링(주기적 요청)** | 가장 단순, 특별한 인프라 불필요. | 실시간성이 떨어지고 요청 수가 많아짐. 순간 대량 접속 상황과 상성이 나쁨. |
| **WebSocket** | 양방향, 범용적. 채팅·협업엔 정석. | 우리는 서버→클라 단방향이면 충분한데, 연결 관리·재연결·하트비트 등 복잡도가 과함. HTTP 인프라(프록시/인증) 재사용도 SSE가 더 쉬움. |
| **SSE** | 단방향 스트리밍에 딱 맞고 표준 HTTP 위에서 동작(EventSource). 자동 재연결 내장. Nginx·인증 흐름과 궁합 좋음. | 단방향뿐(우리 용도엔 문제 없음), 오래된 브라우저/IE 제약(무관). |

## 근거 (Rationale)
요구가 "서버가 순번 변화를 밀어주는 단방향"이므로 SSE가 요구에 정확히 부합한다. WebSocket을 쓰면 배우는 것도 있지만, 이 유스케이스에선 불필요한 복잡도를 지불하는 것. "요구에 맞는 최소 도구를 고른다"는 판단 자체가 시니어 역량이다. (WebSocket은 이후 확장 아이디어로 회고에 남긴다.)

## 결과 (Consequences)
- 표준 HTTP로 구현·배포가 단순, Nginx 프록시 설정도 간단.
- 재연결·백프레셔·연결 정리 같은 스트리밍 운영 이슈를 SSE 범위에서 학습.
