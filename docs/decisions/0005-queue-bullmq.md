# 0005. 큐: BullMQ

- 상태: Accepted
- 날짜: 2026-07-10
- 관련: 0004(Redis)

## 맥락 (Context)
결제·알림은 요청 스레드에서 동기로 처리하면 응답이 느려지고 실패가 전파된다. 결제는 지연·재시도·타임아웃(좌석 홀드 해제)이 필요하므로 비동기 작업 큐가 요구된다.

## 결정 (Decision)
비동기 작업 큐로 **BullMQ**(Redis 기반)를 사용한다.

## 고려한 대안 (Alternatives)
| 대안 | 장점 | 단점 / 채택하지 않은 이유 |
|---|---|---|
| **직접 구현(setTimeout/DB 폴링)** | 의존성 없음. 원리를 밑바닥부터 이해. | 재시도·지연·동시성·가시성을 다 재발명해야 함. 프로덕션 신뢰성이 낮고 1개월 일정에 낭비. |
| **RabbitMQ / Kafka** | 진짜 메시지 브로커. 대규모·이벤트 스트리밍의 정석. | 별도 인프라 운영 부담이 큼. 이번 규모엔 오버엔지니어링. 단, "왜 이게 필요해지는가"는 ADR/회고에 언급해 학습 맥락으로 남긴다. |
| **BullMQ** | 이미 도입한 Redis를 그대로 백엔드로 사용. 재시도·지연·반복·동시성·대시보드(bull-board) 내장. TypeScript 친화적. | 대규모 이벤트 스트리밍(로그 파이프라인 등)엔 부적합 — 하지만 우리 용도(작업 큐)엔 최적. |

## 근거 (Rationale)
이미 Redis가 있으니 인프라 추가 없이 큐를 얻는다. 재시도·지연 작업(결제 타임아웃 후 좌석 반환)·동시성 제어 같은 "큐를 쓰는 진짜 이유"를 최소 설정으로 실습할 수 있다. RabbitMQ/Kafka는 규모가 커질 때의 선택지로 회고에 남겨, 트레이드오프를 아는 것으로 대체한다.

## 결과 (Consequences)
- 결제/알림을 비동기화하고 멱등성 키로 중복을 막는 실습 확보.
- bull-board로 큐 적체를 시각적으로 관측(운영 감각).
