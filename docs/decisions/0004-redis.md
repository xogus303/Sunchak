# 0004. Redis 도입

- 상태: Accepted
- 날짜: 2026-07-10
- 관련: 0005(BullMQ)

## 맥락 (Context)
선착순 예매에는 (1) 순간 대량 요청에서의 원자적 재고 차감, (2) 대기열 순번 관리, (3) 캐싱, (4) 비동기 작업 큐의 저장소가 필요하다. RDB만으로도 일부 가능하지만 경합이 심할 때 병목이 된다.

## 결정 (Decision)
**Redis**를 도입해 원자 연산·대기열·캐시·큐 백엔드로 활용한다.

## 고려한 대안 (Alternatives)
| 대안 | 장점 | 단점 / 채택하지 않은 이유 |
|---|---|---|
| **DB만 사용(락으로 해결)** | 인프라 하나로 단순. | 순간 스파이크에서 행 락 경합이 병목. W2에서 이 한계를 직접 재현한 뒤 Redis로 개선하는 게 학습 서사의 핵심이라, "DB만"은 시작점일 뿐 최종안이 아님. |
| **Memcached** | 캐시엔 단순·빠름. | 원자 연산(`DECR`, Lua), 자료구조(Sorted Set), pub/sub, 큐 백엔드 등 다목적성이 없음. 우리 요구를 못 덮음. |
| **Redis** | 싱글 스레드 기반 원자 연산으로 복잡한 락 없이 정합성 확보. Sorted Set으로 대기열, Lua로 복합 원자 연산, BullMQ 백엔드까지 하나로. | 인메모리라 영속성/장애 시 정합성 설계 주의 필요(학습 포인트). |

## 근거 (Rationale)
Redis의 싱글 스레드 원자성은 "왜 티켓팅이 Redis를 쓰는가"라는 질문의 정답에 가깝다. W2에서 DB 락과 Redis 원자 차감의 처리량 차이를 수치로 비교하고, W3에서 대기열(Sorted Set)·비동기 큐(BullMQ) 백엔드로 재사용한다. 하나의 컴포넌트로 여러 핵심 학습 주제를 관통한다.

## 결과 (Consequences)
- 원자 차감·대기열·캐시·큐를 한 곳에서 실험 가능.
- "인메모리 저장소의 영속성/정합성 트레이드오프"라는 새로운 학습 주제도 함께 떠안는다(의도된 학습).
